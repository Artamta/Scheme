2. Goal and Approach
------------------------------------------------------------

The system models N particles using the Lennard-Jones (LJ 6-12) pair potential:

V(r) = 4*epsilon*((sigma/r)^12 - (sigma/r)^6)

The force is obtained as:

F(r) = -dV/dr

This implementation uses symbolic differentiation from Assignment I to derive the
force expression once, then evaluates it numerically at runtime.

The implementation is intentionally incremental:

1) First understand and validate in 1D.
2) Then move to 2D by replacing scalars with vectors.
3) Then move to 3D with the same simulation engine.

This keeps the code easy to evaluate and shows increasing complexity without
rewriting the physics algorithm.

3. Abstraction Layers
------------------------------------------------------------

Layer A: Symbolic Math Layer
- File: src/model_physis.scm
- Defines symbolic LJ potential expression.
- Uses Assignment I functions deriv and simplify to build symbolic force.

Layer B: Physical Model Layer
- File: src/model_defination.scm
- Defines species masses and pair-dependent LJ parameters for A-A, A-B, B-B.
- Provides parameter lookup helpers (epsilon and sigma per pair).

Layer C: Vector Physics Layer
- File: src/model_physis.scm
- Defines vector add/subtract/scale/norm/unit helpers.
- Computes pair force vectors and net force on each particle.
- Works for 1D, 2D, and 3D by data shape only.

Layer D: Integration Layer
- File: src/model_physis.scm
- Uses Verlet position update:
  r(t+dt) = 2*r(t) - r(t-dt) + a(t)*dt^2
- Uses velocity estimate:
  v(t) = (r(t+dt) - r(t-dt)) / (2*dt)

Layer E: Orchestration and Output Layer
- File: src/main.scm
- Runs simulation loop for multiple time steps.
- Provides run-1d, run-1d-approach, run-2d, run-3d wrappers.
- Prints clear per-step logs.
- Exports CSV trajectories for plotting.

6. Bottlenecks and Complexity
------------------------------------------------------------

Main computational bottleneck:
- Pairwise force loop over all i and j != i.

Time complexity per step:
- O(N^2 * D)
  where N = number of particles, D = dimension (1,2,3).

Measured in code:
- Each step prints directed pair evaluations and step time.

Notes:
- This assignment version prioritizes readability and correctness.
- Future optimization can use neighbor lists or cutoffs.

7. Required Conceptual Questions
------------------------------------------------------------

1) What changes are needed for multi-species systems?

Already partly implemented:
- Each particle has a species label.
- Each species has mass (mass-of).
- LJ parameters are pair-dependent via table:
  (A,A), (A,B), (B,B).

To extend further:
- Add more species labels.
- Expand mass table and LJ pair table.
- No integration logic change needed.

2) What changes are needed for different potentials?

Current design already separates potential/force from integrator.
To switch potential:
- Replace symbolic potential expression in model_physis.scm.
- Recompute force expression via deriv/simplify.
- Keep net-force accumulation and Verlet loop unchanged.

3) How is symbolic differentiation assignment incorporated?

Directly incorporated as follows:
- Assignment I deriv and simplify are loaded.
- LJ potential is represented symbolically.
- Force expression is generated by symbolic differentiation.
- Numeric simulation evaluates that expression for given r, epsilon, sigma.

So Assignment I is reused as a physics-force generator.

4) Comment on process view vs analytical view.

Analytical view:
- Tries to produce closed-form functions for behavior directly.

Computational process view:
- Builds behavior step-by-step in time.
- Uses local updates (force -> acceleration -> position/velocity).
- Produces trajectories by iteration, not closed-form formulas.

In this assignment, Verlet integration clearly demonstrates process view:
state at time t+dt is produced from state at earlier times.


------------------------------------------------------------
8. Pros and Cons of This Approach
------------------------------------------------------------

Pros:
- Easy to read and evaluate.
- One shared engine for 1D, 2D, 3D (low duplication).
- Direct reuse of symbolic differentiation work.
- CSV output supports plotting and analysis.
- Dedicated tests verify correctness basics.

Cons:
- O(N^2) force loop can be expensive for large N.
- No boundary conditions or thermostat.
- No energy-conservation diagnostic yet.
- Cutoff and neighbor-list optimization not included.

- 2D: x1,x2, v1,v2, f1,f2, a1,a2
- 3D: x1,x2,x3, ...

This makes plotting straightforward in Python, gnuplot, or spreadsheets.
